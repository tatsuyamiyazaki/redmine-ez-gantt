redmine_ez_gantt 機能仕様書

Redmine 6 系向けのプロジェクト用ガントチャートプラグイン。バーのドラッグ、進捗編集、親子付け替え、依存関係 (先行/後続) の追加・削除をブラウザ上で直接行えます。

Version 0.0.1 Redmine ≥ 6.0.0 Plugin ID: redmine_ez_gantt Module: easy_gantt

1. 概要

本プラグインは Redmine のプロジェクト画面に「Easy Gantt」メニューを追加し、対象プロジェクトとその子孫プロジェクトに紐づくチケットを 1 つのガントチャートに統合表示します。

表示

サブプロジェクト統合ビュー

選択中プロジェクト + すべての子孫プロジェクトのチケットを 1 枚のチャートに集約します。

編集

ドラッグ操作

バーの中央ドラッグで全体移動、左右端のハンドルで開始日/期限のみを変更できます。

編集

進捗・親付け替え

0〜100% の数値入力での進捗更新と、行のドラッグ&ドロップでの親付け替えに対応します。

依存

依存関係 (precedes)

バー右端のドット (コネクタ) からドラッグで先行・後続関係を作成、矢印クリックで削除します。

可視化

イナズマ線 / クリティカルパス

ツールバーから 2 種類の進捗分析オーバーレイをトグルできます。

UX

楽観的更新

サーバー保存前に UI を先行反映し、エラー時はクライアント側で巻き戻します。

3. 権限とメニュー

プロジェクトモジュール easy_gantt 配下に 2 つの権限を定義しています。プロジェクトでこのモジュールを有効にしないとメニューに表示されません。

キー表示名 (ja)説明
easy_ganttEasy Ganttガントチャートメニューを有効化するモジュール

権限定義

権限付与対象アクション備考
view_easy_gantt index, issues, relations ガント画面の表示・チケット一覧 / 依存関係取得。プロジェクトメンバーであることが必須 (require: :member)。
edit_easy_gantt index, update_issue, update_issue_parent, create_relation, delete_relation 編集系 API の利用と、チケット側プロジェクトを基準とした権限チェック (クロスプロジェクト編集対応)。

メニュー

プロジェクトメニューに、標準ガント (gantt) の直後に挿入されます。

4. HTTP エンドポイント

サーバー側のエンドポイントは画面 1 本 + 取得 2 本 + 編集 4 本の計 7 本です。すべてプロジェクトメンバーまたはチケットのプロジェクトに対する権限チェックが行われます。

共通仕様

  • 認証: Redmine のセッション (credentials: same-origin) と、状態変更系は X-CSRF-Token ヘッダが必須。
  • Content-Type: JSON 受け渡しは application/json。日付は ISO 8601 (YYYY-MM-DD) で送受信。
  • レスポンス成功: {"success": true, ...}。
  • レスポンス失敗: {"success": false, "errors": [...]}。HTTP は 422 Unprocessable Entity または 403 Forbidden / 404 Not Found。
  • 可視性: チケット取得・更新は Issue.visible スコープで保護され、閲覧不可のチケットは存在しないものとして扱われます。
GET /projects/:project_id/easy_gantt ガント画面 (HTML)

Easy Gantt のメイン HTML を返します。view_easy_gantt 権限が必要。テンプレートは <div id="easy-gantt"> 1 つを描画し、後段の URL は data 属性で渡されます。

HTML data 属性

属性用途
data-issues-urlチケット一覧 JSON の取得先
data-issue-update-url-templateチケット更新 URL のテンプレート (__ISSUE_ID__ を置換)
data-relations-url依存関係一覧 JSON の取得先
data-statusesステータス一覧 (フィルタ用) JSON
data-trackersトラッカー一覧 (フィルタ用) JSON
GET /projects/:project_id/easy_gantt/issues チケット一覧 (JSON)

view_easy_gantt 権限が必要。プロジェクトと 子孫プロジェクト の可視チケットを projects.lft, issues.start_date, issues.id 順で返します。

レスポンス (1 件)

{
  "id": 123,
  "subject": "要件定義",
  "tracker":     { "id": 1, "name": "Task" },
  "status":      { "id": 2, "name": "進行中" },
  "assigned_to": { "id": 5, "name": "dev1" },
  "project":     { "id": 8, "name": "計画フェーズ" },
  "project_id": 8,
  "parent_issue_id": null,
  "start_date": "2026-05-01",
  "due_date":   "2026-05-14",
  "done_ratio": 35,
  "editable":   true
}

editable は User.current.allowed_to?(:edit_easy_gantt, issue.project) を都度評価した結果です。クロスプロジェクト集約表示でも、プロジェクトごとに編集可否が変わります。

GET /projects/:project_id/easy_gantt/relations 依存関係一覧 (JSON)

view_easy_gantt 権限が必要。プロジェクト + 子孫の可視チケット同士をつなぐ precedes (先行) 関係のみを返します。両端がフィルタ外であれば除外されます。

[
  { "id": 7, "from_id": 123, "to_id": 124, "delay": 0 }
]
PATCH /easy_gantt/issues/:id チケット更新 (日付 / 進捗)

edit_easy_gantt が チケットの所属プロジェクト に対して必要。許可される属性は意図的に 3 つに限定されています。

リクエストボディ

{
  "issue": {
    "start_date": "2026-05-01",  // 任意 (省略時は変更なし)
    "due_date":   "2026-05-14",  // 任意 (省略時は変更なし)
    "done_ratio": 50              // 任意 (0..100 の整数)
  }
}

バリデーション

  • start_date / due_date は ISO 8601 でパース。不正値は nil として扱う。
  • start_date > due_date の場合は 422 エラー。
  • done_ratio は整数 0〜100 のみ受け付け、範囲外は 422 エラー。

レスポンス (成功)

{
  "success": true,
  "issue":           { /* 更新後の presenter */ },
  "affected_issues": [ /* カスケード移動された後続チケット (precedes) */ ]
}
日付変更が precedes 後続チケットの開始日を侵食する場合、カスケードロジックにより後続チケットを自動シフトし、affected_issues に含めて返します。
PATCH /easy_gantt/issues/:id/parent 親付け替え

edit_easy_gantt が必要。チケットの parent_issue_id のみを変更します。

リクエストボディ

{ "issue": { "parent_issue_id": 200 } }
// null を渡すとルートチケット化

循環チェック

  • 新しい親が自分自身であれば拒否。
  • 新しい親から祖先方向に辿った系列に自分自身が含まれていれば拒否 (子孫を親にできない)。
  • 該当時は 422 + "invalid parent issue"。
POST /easy_gantt/relations 依存関係 (precedes) 作成

先行チケット側 のプロジェクトに対して edit_easy_gantt が必要 (403)。同一チケットへの自己依存は 422 で拒否。delay は常に 0 で作成されます。

リクエストボディ

{ "from_id": 123, "to_id": 124 }

レスポンス (成功)

{
  "success": true,
  "relation": { "id": 7, "from_id": 123, "to_id": 124, "delay": 0 },
  "affected_issues": [ /* from_id のカスケード結果 */ ]
}
DELETE /easy_gantt/relations/:id 依存関係削除

precedes 関係のみが対象。両端のうち先行側のプロジェクトに対して edit_easy_gantt が必要。該当 ID が見つからない、または不可視の場合は 404。

レスポンス (成功)

{ "success": true }

6. UI 機能

画面は単一の #easy-gantt 要素にフルレンダリングされます。レイアウト定数は DAY_WIDTH=24px / ROW_HEIGHT=32px / LEFT_PANE_WIDTH=320px。

ツールバー

  • イナズマ線 トグル — 進捗線オーバーレイの ON/OFF。
  • クリティカルパス トグル — 最遅完了日に至る親子チェーンをハイライト。
  • チケット一覧を表示/閉じる — 左ペインの折りたたみ。
  • ステータス フィルタ — ID 単一選択。
  • トラッカー フィルタ — ID 単一選択。

左ペイン (Issues)

  • プロジェクト見出し行 + チケット行を DFS 順で描画 (orderedIssues)。
  • 親子階層は padding-left: 8 + (depth + 1) * 18px でインデント表現。
  • 各行に #ID, 件名 (チケット詳細へリンク), トラッカーチップ, 進捗入力欄を配置。
  • 親チケットは件名を太字、期限超過チケットは ID 末尾に「!」、件名を赤色。
  • 編集可能行は draggable。マウスダウン位置が進捗入力上ならドラッグ抑止。
  • 左ペインの幅は 220〜560px でドラッグリサイズ可。折りたたみ時はチャートが全幅化。

右ペイン (Timeline)

  • ヘッダは 年/月行 + 日付行の 2 段構成。
  • 日付セルは曜日に応じて色分け: 土曜=青背景、日曜=赤背景、月初=強調。
  • 背景は CSS グラデーションで日単位 / 行単位のグリッドを描画。
  • 本日 (today) の位置に縦の 今日線を表示。
  • 表示範囲はチケット全件の min(start_date)..max(due_date) と本日を含むよう自動拡張。
  • ヘッダがビューポート上端を超えたら、別途固定ツールバーが日付行と横スクロールバーを表示しスクロール同期。

レイアウト定数とレスポンシブ

定数値意味
DAY_WIDTH24px1 日のセル幅
ROW_HEIGHT32px1 行の高さ
LEFT_PANE_WIDTH320px左ペイン既定幅
レスポンシブ (≤900px)左ペイン 280px、バー日付・トラッカーチップ非表示狭幅画面向けの省略表示

7. インタラクション

バーのドラッグ

ピクセル単位の移動量を DAY_WIDTH で割って日数差分に丸めて反映します。start_date > due_date になる操作はリリース時点でロールバックされます。

掴む位置動作変更される属性
バー中央全体スライドstart_date と due_date を同量シフト
左端ハンドル開始日リサイズstart_date のみ
右端ハンドル期限リサイズdue_date のみ

進捗入力

  • <input type="number" min="0" max="100" step="5">。
  • change で PATCH /easy_gantt/issues/:id を呼び出し、楽観的に done_ratio を更新。
  • editable=false のチケットは入力欄が disabled。

親付け替え (Drag & Drop)

  • 編集可能行は draggable=true。ドロップ先候補は他のチケット行、またはチャート上部のルートドロップゾーン。
  • クライアント側で「自分自身」「子孫への移動」を事前検証してエラーメッセージを表示。
  • サーバー側でも循環チェックを行い、422 で拒否される場合があります。
  • 変更前の親 ID を保持し、失敗時に巻き戻し。

依存関係の追加・削除

  • 編集可能なバーをホバーすると 右端のコネクタ (白丸 / 紫枠) が表示される。
  • コネクタを pointerdown してそのままドラッグするとフルスクリーンオーバーレイ上に破線矢印が伸び、別のバー上でドロップすると POST /easy_gantt/relations が走る。
  • 依存矢印は SVG パス (ベジエ曲線) で描画。透明な太線のヒットエリアを重ね、クリック時に window.confirm 経由で DELETE /easy_gantt/relations/:id を呼び出す。

フラッシュメッセージ

画面上部に常駐する .easy-gantt-flash に、info / success / error の 3 段階で進捗・結果を表示します。保存中・成功・失敗の状態はバー側にも is-saving / is-saved / is-error クラスとして反映されます。

8. 可視化機能

バーの種別

子チケット

通常バー

角丸の灰色バーに緑のプログレス。バー内にラベル「作業中」(ステータス名一致時) と進捗 % を表示。

親チケット

親バー

細身のバーに両端のダイヤ型キャップ。親はバー幅・進捗率いずれもサーバー保存値そのまま (子の集計は行わない)。

期限超過

Overdue

本日より due_date が前 かつ 進捗 100% 未満のとき。赤い内枠と影で強調表示。

クリティカル

Critical Path

クリティカルパストグル ON 時、橙色の内枠と橙系プログレスで表示。

編集不可

Readonly

不透明度を下げて表示。ドラッグハンドル・コネクタ非表示。

保存中

Saving / Saved / Error

サーバー往復に合わせて影色が変化。エラー時は赤いグロー後に元状態へ巻き戻し。

クリティカルパス計算

JS 側で計算 (computeCriticalPath)。サーバーには情報を送らず、フィルタ後のチケットツリーから次の手順で導出します。

  1. 各ルートチケットについて、子孫を含めた最遅 due_date を再帰計算。
  2. 全ルートの中で最も遅い日付 globalLatest を求める。
  3. globalLatest に到達するルートを起点に、各階層で「最遅子」を選びながら降下し、通過したチケット ID を criticalIds に蓄積。
仕様上の留意点: このアルゴリズムは依存関係 (precedes) ではなく 親子ツリーの最遅 due_date を見ます。依存関係でクリティカルチェーンを引き出す厳密な CPM ではない点に注意してください。

イナズマ線 (Progress Line)

各チケット行ごとに「現在の進捗が時間軸上どこに相当するか」をドットでプロットし、行間をポリラインで結びます。

  • 進捗位置 = start_date + duration × done_ratio / 100。
  • 進捗位置が 今日線より左かつ done_ratio < 100 なら赤ドット (遅延)。それ以外は緑ドット。
  • 線色は橙 (#f59e0b)、SVG オーバーレイで描画。

依存関係矢印

  • SVG ベジエパスで from.due_date + 1日 右端から to.start_date 左端へ。
  • 矢頭マーカー (id="dep-arrowhead") は紫 (#6366f1)。
  • ヒット用に stroke=transparent, stroke-width=10 のパスを重ね、クリックで削除確認。

9. 依存関係カスケード

先行チケットの日付が変わった、または新しい先行関係が作られたときに、後続チケットの開始日 / 期限を自動的に押し出します。

適用契機

  • PATCH /easy_gantt/issues/:id でチケットを更新し、保存が成功したとき。
  • POST /easy_gantt/relations で先行関係を新規作成し、保存が成功したとき。

計算式

delay         = relation.delay (本プラグインでは作成時 0 固定)
min_start     = predecessor.due_date + delay + 1
if successor.start_date < min_start:
  duration                 = successor.due_date - successor.start_date
  successor.start_date     = min_start
  successor.due_date       = min_start + duration
  affected << successor
  # successor を起点に再帰 (visited Set で循環防止)

挙動の要点

  • 後続チケットの 期間 (duration) を保ったまま シフトする (右側に押し出すだけ)。
  • 後続の start_date が新しい下限 min_start より 後 なら何もしない (前倒しは行わない)。
  • update_columns による直接書き込みのため、コールバックやバリデーションは走らない。
  • 循環は visited Set で抑止。
  • 影響を受けたチケットは affected_issues としてレスポンスで返り、フロントは replaceIssue で UI を更新。
フロント側からの 個別の遅延編集 (delay 変更) API は提供されません。delay を変更したい場合は Redmine 標準の関連編集画面を使用してください。

10. フロント実装の特徴

状態管理

IIFE 内のシングルトン state オブジェクトに全状態を集約。renderGantt() は差分更新せず、毎回フルレンダリングします。

キー用途
issues / issueMapサーバーから取得したチケット配列 (ID -> 参照)
relationsprecedes 関係の配列
pendingSaves / pendingProgressSaves進行中の楽観更新 (saveSeq でレースを判定)
leftPaneWidth / leftPaneCollapsed左ペインの寸法・表示状態
filterStatusId / filterTrackerIdフィルタ状態
showProgressLine / showCriticalPathオーバーレイのトグル
dependencyDrag依存関係ドラッグ中の一時状態
flash最後に表示したメッセージ

楽観的更新

  1. 編集操作を受けてクライアント側で先に state.issues を書き換え再描画。
  2. 同時に pendingSaves.set(issueId, requestId) でリクエスト ID を記録。
  3. サーバーが成功を返したら、レスポンスの issue と affected_issues を replaceIssue で反映。pendingSaves.get(issueId) === requestId でない場合は、より新しい操作に上書きされたとみなして無視。
  4. 失敗時は変更前のチケットをそのまま replaceIssue で復元し、エラーフラッシュを表示。

レンダリング

  • orderedIssues で DFS 順に並べた上で、プロジェクト見出し行を間に挿入 (buildDisplayEntries)。
  • 右ペインのチャート行と左ペインのテキスト行は同じ順序・同じ高さで 1:1 対応。
  • 横スクロールは右ペイン本体と固定ツールバーで双方向同期。
  • レンダリングは Seed の 300 件規模で問題なし。仮想スクロールは未実装。

クライアント側のフェイルセーフ

  • ネットワーク失敗 / 非 OK レスポンスは catch でまとめて受け、エラーフラッシュ + バー赤グロー + 旧状態への replaceIssue で復元。
  • 同一チケットへの連続編集は saveSeq / progressSaveSeq で順序を保証し、古いレスポンスは破棄。
  • 初期ロードに失敗した場合 (fetch が reject) は renderError がメッセージのみを描画。
